# Public roadmap for June 2026

Status: public build roadmap.

Date: 2026 06 16

This roadmap describes the next public infrastructure steps for the Medical AI Failure Atlas and SourceCheckup Medical. It is a builder roadmap, not a claim of clinical validation, regulatory approval, institutional endorsement, model safety, or clinical deployment readiness.

All current public examples are synthetic. No patient data is used.

## Direction

The repository is moving from a static failure atlas into a visible medical AI assurance stack:

1. Failure Atlas entries for repeated medical AI failure modes.
2. SourceCheckup Medical for source discipline and citation review queues.
3. TR MedAI Safety Suite materials for Turkish medical language model evaluation.
4. Clinician review protocols for case review, disagreement, and release gates.
5. Assurance Lab templates for model cards, risk cards, data cards, and audit trails.
6. Health data quality cards for dataset readiness and label review.

## June public targets

1. Publish a larger SourceCheckup v0.2 example set with synthetic answers that cover guideline language, policy language, DOI style locators, URL style locators, and unsupported broad source wording.
2. Add a public issue queue where contributors can submit synthetic failure cases and source review examples.
3. Expand the Failure Atlas taxonomy from a short preview into a reviewable category map.
4. Add a minimal public leaderboard design for future model run reporting without claiming model safety or clinical usefulness.
5. Add Turkish clinician literacy material that can be used as a practical public teaching packet.
6. Add health data quality and label audit templates for future dataset readiness work.

## External visibility sequence

1. Public repo commits that make the work inspectable.
2. Public GitHub issues that invite concrete synthetic contributions.
3. Short public explanations of what each tool checks and what it does not check.
4. Collaboration packets for labs, benchmark maintainers, and Turkish national AI ecosystem routes.
5. Public release notes only after the repo has enough runnable checks and contribution surfaces.

## Safety boundaries

This project will not claim:

1. Clinical advice.
2. Patient data use.
3. Clinical deployment.
4. Clinical validation.
5. Regulatory approval.
6. Institutional endorsement.
7. Official national program acceptance.
8. Model safety proof.

## Next concrete build steps

1. Add SourceCheckup v0.2 synthetic examples and generated reports.
2. Add a Failure Atlas case intake checklist that separates clinical risk, source risk, privacy risk, and communication risk.
3. Add an assurance card template for medical language model releases.
4. Add a Turkish safety benchmark case pack with clear labels and reviewer instructions.
5. Create a public issue for SourceCheckup v0.2 review and contribution.
